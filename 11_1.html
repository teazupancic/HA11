HTTP Methode Beschreibung
GET Die GET-Methode fordert eine Darstellung der angegebenen Ressource an. GET-Requests werden nur dazu verwendet, Daten zu lesen und nicht zu ändern. Wenn sie auf diese Weise verwendet werden, gelten sie als sicher. Das heißt, sie können ohne Risiko der Datenänderung aufgerufen werden. Die Antwort auf eine GET-Anfrage kann zwischengespeichert werden; ein Cache kann es verwenden, um nachfolgende GET und HEAD zu befriedigen.
HEAD Die HEAD-Methode ist gleich wie GET, außer dass der Server keinen Nachrichtentext (body) in der Antwort senden darf. Wie bei einer GET-Methode kann auch eine HEAD-Methode zwischengespeichert werden.
POST Die POST-Methode wird verwendet, um eine Entität an die angegebene Ressource zu senden, was häufig eine Statusänderung oder Nebeneffekte auf dem Server verursacht. Wir erstellen eine untergeordnete Ressource für eine andere (z. B. übergeordnete, parent) Ressource. Beispiele: 1.Bereitstellen von Feldern, die in ein HTML-Formular für einen Datenverarbeitungsprozess eingegeben werden, 2.Senden einer Nachricht an eine Newsgroup, eine Mailingliste, einen Blog oder eine ähnliche Gruppe von Artikeln. 3.Erstellen einer neuen Ressource, die vom Ursprungsserver noch identifiziert werden muss. 4.Daten an die vorhandene Repräsentation einer Ressource anhängen.
PUT Wird verwendet, um eine Repräsentation der ursprünglichen Ressource zu aktualisieren (update). Wenn die Ressource nicht existiert, wird mit PUT eine erstellt. Die Methode ist nicht sicer, weil sie die Daten modifiziert. Put ist nicht im Cache speicherbar.
DELETE Es wird verwendet, um eine durch einen URI identifizierte Ressource zu löschen. Wenn man DELETE für eine Ressource ein zweites Mal aufruft, wird oft eine 404 (nicht gefunden) zurückgegeben. Wenn die Zielressource über eine oder mehrere aktuelle Darstellungen verfügt, werden sie möglicherweise vom Ursprungsserver zerstört oder nicht. Relativ wenige Ressourcen ermöglichen die DELETE-Methode.
CONNECT Die CONNECT-Methode baut einen Tunnel zu dem Server auf, der von der Zielressource identifiziert wird. Tunnel werden häufig verwendet, um eine virtuelle End-to-End-Verbindung über einen oder mehrere Proxies zu erstellen, die dann mithilfe von TLS gesichert werden können. Die meisten Ursprungsserver implementieren CONNECT nicht.
OPTIONS Beschreibt die Kommunikationsoptionen für die Zielressource. Mit dieser Methode kann ein Client die Optionen und / oder Anforderungen ermitteln, die einer Ressource oder den Funktionen eines Servers zugeordnet sind. Ein Server, der eine erfolgreiche Antwort auf OPTIONS generiert, sollte Headerfelder senden, die möglicherweise vom Server implementierte optionale Funktionen anzeigen.
TRACE Die TRACE-Methode fordert eine entfernte Rückmeldung der Anforderungsnachricht auf Anwendungsebene an. Mit TRACE kann der Client sehen, was am anderen Ende der Anforderungskette empfangen wird, und diese Daten für Test- oder Diagnoseinformationen verwenden.


Spezifikation spezielle Eigenschaften Beschreibung Methoden, die die Eigenschaft besitzen
"safe methods" Request-Methoden gelten als "sicher", wenn ihre definierte Semantik schreibgeschützt ist. GET, HEAD, OPTIONS und TRACE
"Idempotent Methods" Die Methode kann mehrmals ohne unterschiedliche Ergebnisse aufgerufen werden. PUT, DELETE + sichere Methoden
"Cacheable Methods" Methoden können als "cachefähig" definiert werden, um anzuzeigen, dass Antworten auf sie für die zukünftige Wiederverwendung gespeichert werden dürfen. GET, HEAD und POST
